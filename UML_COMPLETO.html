CAPÍTULO 1: Fundamentos de UML

1.1 ¿Qué es UML?

UML (Unified Modeling Language) es un lenguaje gráfico estándar para visualizar, especificar, construir y documentar los artefactos de sistemas de software. Fue desarrollado por Grady Booch, James Rumbaugh e Ivar Jacobson (los "tres amigos") y estandarizado por el OMG (Object Management Group) en 1997. La versión vigente es la 2.5.1.

🔑 Idea clave: UML NO es un proceso ni una metodología. Es un lenguaje (tiene sintaxis y semántica) que puede usarse con cualquier metodología (RUP, Scrum, XP, etc.).

Las 4 funciones de UML:

Función Descripción
Visualizar Hacer visible lo invisible: estructuras, comportamientos, relaciones.
Especificar Definir con precisión qué debe hacer el sistema.
Construir Guiar la implementación del código.
Documentar Preservar las decisiones de diseño.

1.2 Diagramas Estructurales vs. de Comportamiento

UML 2.5.1 define 14 diagramas organizados en dos grandes categorías:

Diagramas Estructurales (Estática) Diagramas de Comportamiento (Dinámica)
  • Clases — estructura estática
  • Objetos — instancias en un momento dado
  • Componentes — módulos físicos
  • Despliegue — infraestructura
  • Paquetes — organización lógica
  • Composición estructurada
  • Perfiles
  • Casos de Uso — requisitos funcionales
  • Actividades — flujos de trabajo
  • Estados — ciclo de vida de un objeto
  • Secuencia — interacción en el tiempo
  • Comunicación — interacción entre objetos
  • Timing — restricciones temporales
  • Interacción — visión general
🎯 Regla mnemotécnica:
• Estructurales = "fotografía" del sistema (qué hay)
• Comportamiento = "video" del sistema (qué pasa)

1.3 Modelo Conceptual del UML

Todo diagrama UML se construye con 3 bloques básicos:

  1. Elementos: las "piezas" (clases, objetos, estados, etc.)
  2. Relaciones: cómo se conectan (asociación, herencia, etc.)
  3. Diagramas: la combinación de elementos y relaciones con un propósito

Y se rigen por:

  • Reglas: nombres, visibilidad (+, -, #), integridad, ejecución
  • Mecanismos comunes: adornos, especificaciones, divisiones, extensibilidad (estereotipos <<...>>)

1.4 ¿Cómo elegir el diagrama correcto?

La elección depende de qué quieres comunicar:

¿Qué quiero mostrar? Diagrama recomendado
Requisitos funcionales / qué hace el sistema Casos de Uso
Estructura estática del dominio Clases
Cómo colaboran los objetos en el tiempo Secuencia
Qué objetos colaboran y cómo se conectan Comunicación
Flujo de un proceso / algoritmo Actividades
Ciclo de vida de un objeto Estados
Organización modular del sistema Paquetes
Arquitectura física / módulos desplegables Componentes
Infraestructura / dónde se ejecuta Despliegue

CAPÍTULO 2: Diagrama de Casos de Uso

2.1 Teoría

El Diagrama de Casos de Uso captura los requisitos funcionales del sistema desde la perspectiva del usuario. Muestra qué hace el sistema (no cómo lo hace internamente). Es el primer diagrama que se elabora en un proyecto.

🎯 Propósito: Comunicar con stakeholders no técnicos y definir el alcance del sistema.

2.2 Elementos

Elemento Notación Descripción
Actor 🧍 Muñeco (stick figure) Entidad externa que interactúa con el sistema. Puede ser un usuario, otro sistema o un dispositivo. Se ubica fuera del sistema.
Caso de Uso ⬭ Elipse con nombre Unidad de funcionalidad que produce un resultado observable para un actor. El nombre debe ser un verbo + sustantivo (ej: "Realizar Pedido").
Sistema (Sujeto) ▭ Rectángulo grande Define el alcance del sistema. Los casos de uso van dentro; los actores, fuera.
Asociación — Línea sólida Conecta un actor con un caso de uso que puede ejecutar.

Tipos de Actores:

  • Actor Primario: quien inicia la interacción y recibe el beneficio (ej: Cliente).
  • Actor Secundario: quien participa pero no es el beneficiario principal (ej: Sistema de Pagos externo).

2.3 Relaciones entre Actores y Casos de Uso

2.3.1 Generalización entre Actores

Un actor especializado hereda las relaciones de un actor general.

@startuml left to right direction actor Admin actor Empleado actor Cliente Empleado -up-|> Cliente Admin -up-|> Empleado @enduml

Admin hereda todo lo que puede hacer Empleado, y Empleado hereda lo de Cliente.

2.4 Relaciones entre Casos de Uso

Relación Notación Semántica Ejemplo
<<include>> Flecha punteada → con etiqueta OBLIGATORIA. El caso base SIEMPRE ejecuta el incluido. Factoriza comportamiento común. "Realizar Pedido" include "Validar Stock"
<<extend>> Flecha punteada ← con etiqueta OPCIONAL/CONDICIONAL. Se ejecuta solo si se cumple una condición. Requiere punto de extensión. "Aplicar Cupón" extend "Realizar Pedido" (solo si hay cupón)
Generalización Triángulo hueco △ Un caso de uso hijo hereda y puede especializar al padre. "Pagar con Tarjeta" es un tipo de "Pagar"
⚠️ Regla de oro para el examen:
• <<include>> = "siempre pasa" → la flecha va DESDE el base HACIA el incluido
• <<extend>> = "a veces pasa" → la flecha va DESDE el extensión HACIA el base
¡Las flechas van en direcciones OPUESTAS!

Puntos de Extensión:

Son marcas dentro del caso de uso base donde puede insertarse un comportamiento extendido. Se etiquetan con un nombre (ej: PE1: tiene cupón).

2.5 Especificación de un Caso de Uso

Cada caso de uso debe tener una ficha textual con:

  • Nombre
  • Actor principal
  • Precondiciones (qué debe cumplirse antes)
  • Postcondiciones (qué se garantiza al terminar)
  • Flujo principal (Happy Path)
  • Flujos alternativos
  • Excepciones

2.6 Ejemplo Completo

@startuml left to right direction actor Socio actor Bibliotecario rectangle "SISTEMA BIBLIOTECA" { usecase "Préstamo de Libro" as prestamo usecase "Verificar Socio" as verificar usecase "Reservar Libro" as reservar usecase "Notificar Disponibilidad" as notificar } Socio --> prestamo Bibliotecario --> prestamo prestamo ..> verificar : <<'include'>> notificar ..> reservar : <<'extend'>> [si hay espera] @enduml

2.7 Errores Frecuentes

❌ Errores típicos:
  1. Invertir las flechas de include y extend.
  2. Poner actores dentro del sistema.
  3. Usar casos de uso demasiado técnicos ("Actualizar BD") en vez de funcionales.
  4. Conectar actores entre sí con líneas (solo se conectan con casos de uso).
  5. Hacer diagramas con cientos de casos de uso sin agrupar.
  6. Confundir include (obligatorio) con extend (opcional).
  7. Olvidar la especificación textual del caso de uso.

2.8 Ejercicios de Casos de Uso

CAPÍTULO 3: Diagrama de Clases

3.1 Teoría

El Diagrama de Clases es el diagrama más importante de UML. Muestra la estructura estática del sistema: clases, atributos, operaciones y las relaciones entre ellas. Es la base del diseño orientado a objetos.

3.2 La Clase

Una clase es una plantilla que define atributos y operaciones comunes a un conjunto de objetos.

@startuml hide circle skinparam classAttributeIconSize 0 class Estudiante { - id: int - nombre: String - email: String -- + inscribirse(): void + buscar(): Estudiante } @enduml

Estructura de un atributo/operación:

visibilidad nombre : tipo [= valor]

3.3 Visibilidad

Símbolo Tipo Acceso
+ Público Cualquier clase puede acceder
- Privado Solo la propia clase
# Protegido La clase y sus subclases
~ Paquete Clases del mismo paquete

3.4 Clases Abstractas e Interfaces

Tipo Notación Características
Clase Abstracta Nombre en cursiva o con estereotipo <<abstract>> No puede instanciarse. Sirve como plantilla para herencia.
Interfaz Círculo (lollipop) o rectángulo con <<interface>> Contrato: solo declara operaciones, sin implementación. Las clases la "realizan".

3.5 Objetos

Un objeto es una instancia concreta de una clase, con valores específicos.

@startuml object "estudiante1 : Estudiante" as estudiante1 { id = 12345 nombre = "Juan Pérez" email = "juan@uni.edu" } @enduml

Nota: el nombre del objeto va en minúsculas, seguido de : y el nombre de la clase.

3.6 Relaciones (de más débil a más fuerte)

🔑 Truco para el examen: Mirá primero la punta de flecha:
• Triángulo = herencia/realización
• Rombo = agregación/composición
• Flecha simple = dependencia
• Línea sin flecha = asociación
Luego mirá si la línea es sólida o punteada.
Relación Notación Semántica Ejemplo
1. Dependencia ⇢ Línea punteada + flecha abierta "usa temporalmente". La más débil. Cambio en una afecta a la otra. Factura usa CalculadoraImpuestos
2. Asociación — Línea sólida (con multiplicidad) Conexión estructural persistente entre clases. Estudiante — Curso
3. Agregación ◇— Rombo vacío en el lado del "todo" "tiene-un". Todo-parte débil. Las partes pueden existir sin el todo. Departamento ◇— Profesor
4. Composición ◆— Rombo relleno en el lado del "todo" "parte-de". Todo-parte fuerte. Si muere el todo, mueren las partes. Casa ◆— Habitación
5. Generalización —▷ Línea sólida + triángulo hueco Herencia "es-un". Subclase hereda de superclase. Perro —▷ Animal
6. Realización ⇢▷ Línea punteada + triángulo hueco Una clase implementa una interfaz. ServicioPago ⇢▷ IPago

3.7 Multiplicidades

Indican cuántas instancias de una clase pueden relacionarse con una instancia de la otra.

Multiplicidad Significado
1 Exactamente uno
0..1 Cero o uno (opcional)
* o 0..* Cero o más (cualquier cantidad)
1..* Uno o más (al menos uno)
0..5 Entre cero y cinco
2..4 Entre dos y cuatro

Ejemplo con multiplicidades:

@startuml hide circle skinparam classAttributeIconSize 0 left to right direction class Estudiante class Curso Estudiante "1" -- "*" Curso @enduml

3.8 Relaciones Especiales

Clase Asociativa:

Cuando una asociación tiene atributos propios (ej: "Inscripción" entre Estudiante y Curso con fecha y nota).

Asociación N-aria:

Relación entre 3 o más clases simultáneamente, representada con un rombo central.

Atributos de la Asociación:

Se pueden agregar roles, multiplicidades y atributos a las asociaciones.

3.9 Ejemplo Completo

@startuml hide circle skinparam classAttributeIconSize 0 left to right direction abstract class Persona { # nombre # dni -- + getNombre() } class Estudiante { - legajo -- + cursar() } class Curso { - codigo - descripcion -- + getCodigo() } Estudiante -up-|> Persona Estudiante "*" -right- "1" Curso : se inscribe en > @enduml

CAPÍTULO 4: Diagramas de Interacción

Los diagramas de interacción muestran cómo colaboran los objetos para realizar un comportamiento. UML define dos tipos principales: Secuencia y Comunicación. Ambos representan los mismos mensajes, pero con énfasis distintos.

4.1 Diagrama de Secuencia

Características:

  • Énfasis en el ORDEN TEMPORAL de los mensajes
  • Eje vertical = tiempo (hacia abajo)
  • Líneas de vida verticales para cada objeto/participante
  • Mensajes horizontales entre líneas de vida
  • Barras de activación (rectángulos delgados) sobre las líneas de vida

Elementos:

Elemento Notación Descripción
Participante Rectángulo arriba con línea de vida Puede ser un objeto, clase o actor
Línea de vida Línea vertical punteada Representa la existencia del objeto en el tiempo
Mensaje síncrono Flecha sólida con punta llena → El emisor espera la respuesta
Mensaje asíncrono Flecha sólida con punta abierta ⇢ El emisor no espera
Mensaje de retorno Flecha punteada ⇠ Respuesta al mensaje
Activación Rectángulo delgado sobre línea de vida Período en que el objeto está procesando
Fragmento combinado Rectángulo con operador (alt, opt, loop, par) Condicionales, bucles, paralelismo

Operadores de Fragmentos:

Operador Significado Uso
alt Alternativa (if/else) Solo una rama se ejecuta
opt Opcional (if) Se ejecuta o no
loop Bucle Se repite con condición
par Paralelo Se ejecuta simultáneamente
break Interrupción Si se cumple, sale del fragmento

Ejemplo: Retirar dinero de cajero

@startuml actor Usuario participant Cajero participant Banco Usuario -> Cajero : insertar Cajero -> Banco : validar Banco --> Cajero : ok Usuario -> Cajero : pedir $ Cajero -> Banco : debitar Banco --> Cajero : ok Cajero --> Usuario : entregar $ @enduml

4.2 Diagrama de Comunicación

Características:

  • Énfasis en la ESTRUCTURA y las relaciones entre objetos
  • Los objetos se muestran como una red conectada
  • Los mensajes se numeran (1, 2, 3...)
  • No hay eje temporal explícito: el orden se deduce de la numeración
  • Números anidados (1.1, 1.2) indican sub-mensajes

Ejemplo equivalente:

@startuml object Usuario object Cajero object Banco Usuario -> Cajero : 1: insertarTarjeta() Cajero -> Usuario : 2: pedirMonto() Usuario -> Cajero : 3: ingresarMonto() Cajero -> Banco : 4: validar() Banco -> Cajero : 5: ok Cajero -> Banco : 6: debitar() Cajero -> Usuario : 7: entregarDinero() @enduml

4.3 Comparación: Secuencia vs Comunicación

Aspecto Secuencia Comunicación
Énfasis Tiempo / orden Estructura / relaciones
Eje temporal Explícito (vertical) Implícito (números)
Legibilidad temporal Muy clara Requiere seguir números
Legibilidad estructural Menos clara Muy clara (red de objetos)
Uso preferente Cuando importa el orden Cuando importa quién se conecta con quién
Son equivalentes ✅ Sí, se pueden convertir uno en el otro

CAPÍTULO 5: Diagrama de Actividades

5.1 Teoría

El Diagrama de Actividades modela el flujo de trabajo o proceso de negocio. Es similar a un flowchart pero más potente: permite modelar concurrencia, decisiones complejas y responsabilidades de distintos actores.

🎯 Diferencia clave con Estados:
• Actividades = flujo de proceso / pasos secuenciales (qué se hace)
• Estados = ciclo de vida de UN objeto reaccionando a eventos (en qué situación está)

5.2 Elementos

Elemento Notación Descripción
Nodo inicial ● Círculo negro relleno Punto de partida del flujo
Nodo final ⊙ Círculo negro con borde Fin del flujo (o de una rama)
Acción / Actividad ▢ Rectángulo con bordes redondeados Una tarea o paso atómico
Flujo de control → Flecha sólida Indica el orden de ejecución
Decisión ◇ Rombo Bifurcación condicional (if/else)
Fusión (merge) ◇ Rombo Une ramas alternativas
Bifurcación (fork) ▬ Barra negra gruesa Divide el flujo en paralelo
Unión (join) ▬ Barra negra gruesa Sincroniza flujos paralelos
Swimlane Carril vertical u horizontal Indica quién ejecuta la acción
Objeto Rectángulo subrayado Dato que fluye entre actividades

5.3 Swimlanes (Particiones)

Dividen el diagrama en carriles. Cada carril representa un responsable: un actor, una clase, un departamento o un sistema.

@startuml |Cliente| start :Seleccionar productos; |Sistema| :Confirmar pedido; :Solicitar pago; |Banco| :Procesar pago; |Sistema| :Autorizado; |Cliente| :Recibir producto; stop @enduml

5.4 Decisiones (if/else)

El rombo de decisión tiene una entrada y varias salidas, cada una con una guarda (condición entre corchetes).

@startuml start :Iniciar sesión; if (credencial) then (OK) :Acceder; else (INVÁLIDA) :Mostrar error; endif stop @enduml

5.5 Fork / Join (Paralelismo)

Permiten ejecutar actividades simultáneamente.

@startuml start :Preparar pedido; fork :Empaquetar; fork again :Emitir factura; end fork :Enviar; stop @enduml
⚠️ Regla importante: Cada fork debe tener un join correspondiente. No pueden quedar flujos paralelos "sueltos".

5.6 Ejemplo Completo: Proceso de Compra Online

@startuml |Cliente| start :Buscar producto; |Tienda| :Verificar stock; |Stock| if (¿Hay stock?) then (Sí) :Reservar; |Pago| :Procesar; :Autorizar; |Cliente| :Recibir producto; else (No) |Tienda| :Mostrar agotado; endif |Cliente| stop @enduml

CAPÍTULO 6: Diagrama de Estados (Máquina de Estados)

6.1 Teoría

El Diagrama de Estados (o Máquina de Estados) modela el ciclo de vida de un objeto individual. Muestra cómo un objeto cambia de estado en respuesta a eventos.

🎯 ¿Cuándo usarlo? Para objetos con ciclos de vida complejos: Pedidos, Facturas, Cuentas, Ascensores, Semáforos, etc.

6.2 Elementos

Elemento Notación Descripción
Estado inicial ● Círculo negro Punto de partida
Estado final ⊙ Círculo con borde Fin del ciclo de vida
Estado (simple) ▢ Rectángulo con bordes redondeados Situación en la vida del objeto
Estado compuesto Rectángulo con subestados dentro Contiene otros estados (jerarquía)
Transición → Flecha Cambio de un estado a otro
Evento Etiqueta en la flecha Dispara la transición
Guarda [condición] Condición que debe cumplirse
Efecto / acción Acción que se ejecuta al transicionar

6.3 Sintaxis de una Transición

evento [guarda] / efecto

Ejemplo: pagar [monto > 0] / actualizarSaldo()

  • evento: qué dispara (ej: "pagar", "click", "timeout")
  • [guarda]: condición booleana opcional
  • / efecto: acción que se ejecuta

6.4 Actividades Internas del Estado

Dentro de un estado pueden definirse actividades:

Actividad Cuándo se ejecuta
entry / acción Al entrar al estado
exit / acción Al salir del estado
do / acción Mientras se está en el estado (actividad continua)
@startuml state Activo { Activo : entry / enviarBienvenida() Activo : do / monitorearUso() Activo : exit / enviarDespedida() } @enduml

6.5 Regiones Ortogonales

Permiten modelar comportamientos paralelos e independientes dentro de un mismo objeto.

Ejemplo: Un Teléfono puede estar simultáneamente en un estado de "Conectividad" y en otro de "Batería":

@startuml state "Teléfono Encendido" as TelefonoEncendido { [*] --> Conectado Conectado --> Desconectado : desconectar -- [*] --> Cargando Cargando --> al100 : lleno state "Al 100%" as al100 } @enduml

6.6 Ejemplo: Ciclo de Vida de un Pedido

@startuml [*] --> Creado Creado --> Confirmado : confirmar Confirmado --> Pagado : pagar Pagado --> Enviado : enviar Enviado --> Entregado : recibir Entregado --> [*] Confirmado --> Cancelado : cancelar [no enviado] Cancelado --> [*] @enduml

6.7 Ejemplo: Ascensor

@startuml [*] --> Parado Parado --> Subiendo : llamar Subiendo --> Parado : llegar Parado --> Bajando : llamar Bajando --> Parado : llegar @enduml

CAPÍTULO 7: Diagrama de Paquetes

7.1 Teoría

El Diagrama de Paquetes organiza los elementos del modelo en unidades lógicas (paquetes). Es fundamental para gestionar la complejidad de sistemas grandes y mostrar la arquitectura en capas.

🎯 Analogía: Si el sistema es una ciudad, los paquetes son los barrios/distritos. Cada uno tiene su propio espacio de nombres.

7.2 Elementos

Elemento Notación Descripción
Paquete 📁 Carpeta con pestaña Agrupa clases, casos de uso u otros paquetes
Elemento interno [Componente] o clase Clase o componente visible dentro del paquete (opcional en vistas de alto nivel)
Dependencia ⇢ Flecha punteada Un paquete usa elementos de otro
Package Import ⇢ Flecha punteada con <<import>> Importa elementos públicos
Package Access ⇢ Flecha punteada con <<access>> Accede manteniendo visibilidad original
Package Merge ⇢ Flecha punteada con <<merge>> Combina contenido de dos paquetes

7.3 Principios de Diseño Modular

Principio Objetivo Descripción
Alta Cohesión Dentro del paquete Los elementos deben estar fuertemente relacionados. "Lo que cambia junto, va junto".
Bajo Acoplamiento Entre paquetes Las dependencias deben ser mínimas. Un cambio en un paquete no debería romper muchos otros.

7.5 Errores Frecuentes

  • Ciclos de dependencia: Paquete A depende de B y B depende de A. ¡Prohibido!
  • Paquetes con baja cohesión (mezcla responsabilidades).
  • Demasiadas dependencias cruzadas entre paquetes del mismo nivel.

CAPÍTULO 8: Diagrama de Componentes

8.1 Teoría

El Diagrama de Componentes modela la organización física y lógica de los componentes de software. Un componente es una unidad modular, reemplazable y encapsulada que expone interfaces.

🎯 Analogía: Como piezas de un computador (RAM, CPU, GPU) con conectores estándar. Podés cambiar una pieza sin tocar las otras, siempre que respetes la interfaz.

8.2 Elementos

Elemento Notación Descripción
Componente ▢ Rectángulo con icono ◽️ en esquina Unidad modular de software (biblioteca, servicio, módulo)
Interfaz proporcionada ⚪ Círculo (lollipop) Contrato público: servicios que el componente ofrece a su entorno
Interfaz requerida ◗ Semicírculo (socket) Dependencia explícita: servicios que el componente necesita de otros
Puerto ▭ Cuadrado en el borde Punto explícito de interacción (HTTP, eventos, integración con otro subsistema)
Conector de ensamblaje — Línea sólida Conecta lollipop con socket
Conector de delegación ⇢ Línea punteada Delega de un puerto externo a un componente interno

8.3 Relación con otros diagramas

  • vs. Paquetes: Los paquetes son agrupaciones lógicas; los componentes son unidades más concretas (físicas o de despliegue).
  • vs. Despliegue: Los componentes se "montan" sobre nodos en el diagrama de despliegue.
  • vs. Clases: Un componente puede contener varias clases.

8.4 Ejemplo: arquitectura por capas

Una arquitectura por capas organiza componentes según responsabilidades diferenciadas (presentación, aplicación, dominio, infraestructura, persistencia). El diagrama de componentes muestra qué dependencias están permitidas entre capas. El valor de las capas está en imponer dirección y disciplina a las dependencias: si cualquier componente puede depender de cualquier otro, la arquitectura pierde capacidad de orientación.

@startuml left to right direction component "Interfaz Web" as UI component "API Aplicación" as API component "Dominio" as Dominio component "Persistencia" as Persistencia database "BD" as BD UI ..> API API ..> Dominio API ..> Persistencia Persistencia ..> BD @enduml

Las dependencias se orientan desde la interfaz hacia aplicación, dominio, persistencia y base de datos. Una arquitectura por capas no debe evaluarse sólo por la posición visual de los componentes, sino por la dirección real de sus dependencias, la estabilidad de sus interfaces y la capacidad de modificar una capa sin alterar innecesariamente las demás.

Fuente adaptada: Arquitectura de Componentes y Capas (egrpy.org)

8.5 Ejemplo: interfaces proporcionadas y requeridas

Una interfaz proporcionada describe servicios que un componente ofrece a su entorno (una promesa pública). Una interfaz requerida describe servicios que el componente necesita de otros para cumplir su responsabilidad (dependencia explícita). Esta distinción permite representar contratos de colaboración sin exponer detalles internos de implementación.

@startuml left to right direction component "Servicio de Pedidos" as Pedidos component "Servicio de Pago" as Pagos interface "IPedidos" as IPedidos interface "IPagos" as IPagos Pedidos - IPedidos : proporciona Pedidos ..> IPagos : requiere Pagos - IPagos : proporciona @enduml

Servicio de Pedidos proporciona IPedidos y requiere IPagos. Servicio de Pago proporciona el contrato que Pedidos necesita. Modelar interfaces reduce acoplamiento porque los componentes dependen de contratos más estables y no de detalles internos. También favorece reemplazo, prueba, integración incremental y asignación de responsabilidades entre equipos.

Fuente adaptada: Interfaz Proporcionada y Requerida en Componentes UML (egrpy.org)

8.6 Ejemplo: puertos UML

Un puerto representa un punto explícito de interacción entre un componente y su entorno. Permite precisar por dónde se ofrecen o requieren interfaces, especialmente cuando un componente tiene varios canales de comunicación (API HTTP, eventos, integración con otro subsistema).

@startuml left to right direction component "API de Pedidos" as API { portin "HTTP" as HTTP portout "Eventos" as Eventos } interface "IPedidos" as IPedidos interface "IEventosPedido" as IEventos IPedidos -- HTTP Eventos -- IEventos @enduml

API de Pedidos expone un puerto HTTP para recibir solicitudes y un puerto Eventos para publicar cambios relevantes. Los puertos son útiles cuando el componente tiene fronteras de interacción diferenciadas. Si el componente sólo ofrece una interfaz simple y no hay ambigüedad, puede bastar con representar la interfaz sin puerto explícito.

Fuente adaptada: Puerto UML en Diagramas de Componentes (egrpy.org)

8.7 Ejemplo: aplicación web multicapa

Este ejemplo representa una aplicación web multicapa con navegador, frontend, API Gateway, servicios de dominio, bases de datos y broker de eventos. Muestra cómo el diagrama de componentes comunica la integración entre unidades implementables en una arquitectura moderna, sin entrar al detalle de clases.

@startuml component "Navegador" as Browser component "Frontend SPA" as SPA component "API Gateway" as Gateway component "Servicio Usuarios" as Usuarios component "Servicio Catálogo" as Catalogo component "Servicio Pedidos" as Pedidos component "Broker de Eventos" as Broker component "Base Usuarios" as DBU component "Base Pedidos" as DBP Browser ..> SPA : HTTP SPA ..> Gateway : REST/JSON Gateway ..> Usuarios Gateway ..> Catalogo Gateway ..> Pedidos Usuarios ..> DBU Pedidos ..> DBP Pedidos ..> Broker : publica EventoPedido Catalogo ..> Broker : consume actualización @enduml

Lectura del ejemplo

CAPÍTULO 9: Diagrama de Despliegue

9.1 Teoría

El Diagrama de Despliegue muestra la arquitectura de ejecución del sistema: dónde se ejecutan los artefactos, en qué nodos, y cómo se comunican.

🎯 Analogía: Es el "plano de infraestructura operativa". No muestra el diseño lógico, sino dónde se instala físicamente cada cosa.

9.2 Elementos

Elemento Notación Descripción
Nodo ▣ Cubo 3D Recurso computacional (servidor, dispositivo, contenedor)
Dispositivo Cubo 3D con <<device>> Recurso físico (hardware)
Entorno de ejecución Cubo 3D con <<executionEnvironment>> Software que aloja artefactos (JVM, servidor web, contenedor)
Artefacto 📄 Rectángulo con <<artifact>> Archivo físico (JAR, EXE, imagen Docker, script)
Relación de despliegue ⇢ Flecha punteada con <<deploy>> Artefacto se instala en un nodo
Manifestación ⇢ Flecha punteada con <<manifest>> Un componente se materializa en un artefacto
Ruta de comunicación — Línea sólida con protocolo Conexión entre nodos (HTTP, TCP, JDBC)

9.3 Ejemplo: despliegue de arquitecturas en capas

Una arquitectura en capas puede desplegarse en nodos separados o en un mismo entorno de ejecución, según necesidades de seguridad, rendimiento, escalabilidad y mantenimiento. El diagrama de despliegue permite mostrar esa distribución operativa: qué artefactos viven en cada servidor y cómo se comunican en tiempo de ejecución.

@startuml left to right direction node "Cliente" as Cliente node "Servidor Web" as Web { artifact "frontend" as Front } node "Servidor Aplicación" as App { artifact "api" as API artifact "dominio" as Dominio } node "Servidor Datos" as Datos { artifact "BD" as BD } Cliente -- Web : HTTPS Web -- App : REST App -- Datos : SQL @enduml

Lectura del ejemplo

  • Cliente: navegador o dispositivo del usuario final.
  • Servidor Web: aloja el artefacto frontend (HTML, CSS, JavaScript).
  • Servidor Aplicación: contiene los artefactos api y dominio (capa de aplicación y lógica de negocio).
  • Servidor Datos: aloja el artefacto BD (base de datos).
  • Las rutas de comunicación usan protocolos concretos: HTTPS (cliente → web), REST (web → aplicación) y SQL (aplicación → datos).

La separación entre cliente, servidor web, servidor de aplicación y servidor de datos refuerza fronteras operativas y de seguridad. Cada capa lógica del diagrama de componentes (Capítulo 8) puede mapearse a un nodo físico distinto.

💡 Criterio arquitectónico: Separar capas en distintos nodos puede mejorar seguridad y escalabilidad, pero también introduce latencia, complejidad operacional y costos de comunicación. El despliegue debe justificarse por atributos de calidad, no sólo por una separación visual elegante.

  1. FALSO. UML es un LENGUAJE, no una metodología. Tiene sintaxis y semántica, pero no prescribe un proceso.

🔑 REGLAS DE ORO

  1. Include vs Extend: Include = SIEMPRE (flecha →); Extend = A VECES (flecha ←).
  2. Agregación vs Composición: Agregación = débil (◇ vacío); Composición = fuerte (◆ relleno).
  3. Secuencia vs Comunicación: Secuencia = tiempo; Comunicación = estructura.
  4. Actividades vs Estados: Actividades = proceso; Estados = ciclo de vida de UN objeto.
  5. Visibilidad: + público, - privado, # protegido, ~ paquete.
  6. Multiplicidades: 1, 0..1, *, 1..*, 0..n.

  • Si ves muñecos → Casos de Uso.
  • Si ves rectángulos con 3 partes → Clases.

  • Si ves rombos con condiciones → Actividades.
  • Si ves cubos 3D → Despliegue.
  • Si ves carpetas → Paquetes.
  • Si ves lollipops → Componentes.

Diagrama de Clases

🎯 Objetivo: representar la estructura estática del dominio: las clases, sus atributos y operaciones, y las relaciones entre ellas. Es uno de los núcleos del modelado orientado a objetos y la columna vertebral del diseño.
🛠️ Para qué se usa:
  • Modelar el dominio del problema (entidades y sus vínculos) durante análisis y diseño.
  • Servir de base para generar el esquema de base de datos y el código de las clases.

Elemento Descripción

<<include>> Comportamiento obligatorio reutilizado por otro caso de uso.
<<extend>> Comportamiento opcional que se agrega bajo cierta condición.
🔗 Cómo se relaciona con los demás: es el origen del análisis. Cada caso de uso se detalla luego con diagramas de Secuencia o Actividades (cómo se ejecuta), mientras que las entidades que menciona se modelan en el Diagrama de Clases.

Diagrama de Secuencia

🎯 Objetivo: responder por el orden exacto de los mensajes entre los participantes, organizando la interacción sobre un eje temporal descendente. Resalta la temporalidad y la secuencia de colaboración.
🛠️ Para qué se usa:
  • Detallar el flujo de un escenario concreto (un caso de uso).
  • Diseño de interacciones entre objetos/subsistemas.
  • Especialmente útil en diseño y pruebas.

Componentes

Elemento Descripción
Actor / Objeto participante Se ubica en la parte superior; representa quién colabora.
Línea de vida Línea vertical punteada que representa el tiempo del participante.
Foco de control (activación) Rectángulo sobre la línea de vida: periodo en que el objeto está activo.
Mensaje síncrono / retorno Flecha sólida (llamada) y flecha punteada (respuesta).
🔗 Cómo se relaciona con los demás: concreta un caso de uso a nivel de interacción; sus participantes son objetos (instancias de clases del Diagrama de Clases). Es equivalente al Diagrama de Comunicación: ambos muestran la misma interacción, pero uno enfatiza el tiempo y el otro la estructura.

🎯 Objetivo: modelar las interacciones entre objetos enfatizando los vínculos (topología de colaboración) más que la cronología. Conserva el orden de los mensajes mediante numeración.
🛠️ Para qué se usa:
  • Ver la red de relaciones entre objetos que colaboran.

Objeto Participante mostrado como nodo de la red.

Dinámico

Diagrama de Actividades

🎯 Objetivo: modelar procesos, decisiones y bifurcaciones. Es cercano al flujograma pero con semántica UML; abstrae el flujo de trabajo y vuelve visibles las decisiones y los caminos alternativos.
🛠️ Para qué se usa:
  • Detallar el comportamiento interno de un proceso o caso de uso.
  • Modelar flujos de negocio y responsabilidades (con swimlanes).
  • Representar paralelismo (fork/join) y ramificaciones.

Componentes

Elemento Descripción
Nodo inicial / final Círculo relleno (inicio) y círculo con borde (fin).
Acción / Actividad Rectángulo redondeado con la tarea a realizar.
Decisión / Merge Rombo: bifurcación condicional (if/else) y reunión de caminos.
Fork / Join Barra: divide o sincroniza flujos paralelos.
Swimlane (partición) Calle que asigna cada acción a un responsable.
🔗 Cómo se relaciona con los demás: detalla el flujo interno de un caso de uso (alternativa al de Secuencia cuando importa la lógica del proceso más que el intercambio de mensajes). Las acciones suelen corresponder a operaciones de clases del Diagrama de Clases.

Diagrama de Estados

🎯 Objetivo: describir el ciclo de vida de un objeto: sus estados, los eventos que provocan cambios y las transiciones. Muestra cómo cambia un objeto cuando ocurren eventos significativos.
🛠️ Para qué se usa:
  • Modelar objetos cuyo comportamiento depende de su historia previa.
  • Especificar reglas de transición (qué eventos son válidos en cada estado).
  • Diseñar controladores, protocolos y máquinas de estado.

Componentes

Elemento Descripción
Estado inicial / final Círculo relleno ([*] de entrada) y círculo con anillo (fin).
Estado Rectángulo redondeado: situación del objeto en un momento dado.
Transición Flecha etiquetada con evento [guarda] / acción.
Regiones ortogonales Estados concurrentes dentro de un mismo objeto.
🔗 Cómo se relaciona con los demás: se centra en una sola clase del Diagrama de Clases (típicamente una con estados relevantes, como Préstamo o Pedido). Complementa a Secuencia/Actividades: mientras estos muestran interacciones o flujos globales, Estados muestra la evolución interna de un objeto.

Ejercicio 2.1 — Delivery de comida

  • Pedido: id, fecha, estado, total. Método: calcularTotal().
  • Cliente: id, nombre, direccion. Método: realizarPedido().
  • Repartidor: id, nombre, vehiculo.
  • ItemPedido: cantidad, precioUnitario.

Consigna: PlantUML con visibilidad, multiplicidades y tipo correcto de relación (asociación/agregación/composición/herencia). hide stereotype.
